# Core Discipline: Skill Obedience

## The Absolute Rule
**If there is even a 1% chance a skill might apply to your task, you ABSOLUTELY MUST invoke and follow the skill BEFORE taking action.**

This is not negotiable. This is not optional. You cannot rationalize your way out of this.

## When to Use
ALWAYS. Use this mindset at the beginning of every conversation, task, or question.

## Execution Priority (Order of Operations)
When processing a user request, sequence your skills:
1. **Planning/Process Skills First:** (e.g., `brainstorming-designs.md`, `qa-tdd-architect.md`). These dictate HOW you approach the problem.
2. **Architecture Skills Second:** (e.g., `architecture-python-backend.md`, `architecture-frontend-design.md`, `architecture-database-design.md`). These protect the structure.
3. **Tech-Stack Skills Third:** (e.g., `fastapi-backend-patterns.md`, `nextjs-heroui-frontend.md`). These provide the exact implementation details.

## Kill These Rationalizations Immediately (Red Flags)
If you think any of the following, STOP and load the relevant skill:
- ❌ *"This is just a simple code change."* (Simple things break. Use TDD/QA skills.)
- ❌ *"I'll just fix this one bug first, then look at the skill."* (Use the systematic debugging skill BEFORE touching code.)
- ❌ *"I know how React works, I don't need to read the guideline."* (Knowing the concept ≠ using the specific project guidelines.)
- ❌ *"I'll verify it manually."* (Ad-hoc testing is forbidden. Use TDD.)

## The RARV Execution Loop Check
At every step (Gate 4), ensure you have:
1. **PRE-ACT:** Read the relevant skill checklist.
2. **ACT:** Implemented exactly according to the skill.
3. **VERIFY:** Tested against the skill's criteria.
